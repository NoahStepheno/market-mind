## Successes

1. **Infrastructure-first strategy was correct.** Story 1.0's unified error framework + `@market/utils` shared package eliminated repeated negotiation across subsequent stories. Every story after 1.0 directly consumed these foundations.

2. **100% story completion with zero production incidents.** No descoping, no rollbacks. Brownfield project with established patterns made this achievable.

3. **Testing rhythm established early.** 58+ tests co-located with source files. Error handler tests, auth unit tests, 401 retry tests, navigation component tests. Test patterns (mock strategy, renderToString, relative imports in test files) are now established and reusable.

4. **Design system foundation complete.** All 13 color tokens, 12 typography scales, 4 radius levels, 8 spacing levels mapped to TailwindCSS. Global navigation with responsive breakpoints (desktop top bar / mobile bottom tab bar) delivered. Epic 2's UI components have a solid base.

5. **Review process was thorough.** 60 findings across 4 stories — the review caught real issues (Dockerfile build context, non-null assertions, selector stability, aria-current, focus-visible, duplicate API calls on logout).

---

## Challenges

1. **Deferred work accumulation without tracking mechanism.** 30+ deferred items across 4 stories with no systematic tracking. Items sit in story files and are invisible to subsequent development. ~80% are pre-existing issues discovered during development, but pre-existing does not mean not-our-problem.

2. **Testing infrastructure friction.** Almost every story hit test configuration issues: `@/` path alias not resolving in vitest, `render()` failing with "document is not defined" in happy-dom. These are one-time discoveries but each one cost development time.

3. **Brownfield discovery overhead.** Each story's dev notes contain extensive "What Already Exists" sections — Story 1.1's occupies ~30% of the document. Necessary for brownfield work but high discovery cost.

4. **Dev notes knowledge duplication.** Better-Auth deviation documented in Stories 1.1, 1.2, and 1.3. Testing patterns repeated. No shared knowledge source for implementation learnings — each story re-documents context.

5. **Type alignment lag.** Story 1.0 changed error framework (code: number → string), Story 1.2 fixed frontend types to match. Cross-cutting type changes not propagated immediately.

---

## Key Insights

1. **Deferred work needs a tracking mechanism with ownership and timing.** All deferred items — including pre-existing — belong to the team. Every item needs classification (patch/design/upstream), risk level, and a suggested resolution epic.

2. **Cross-story knowledge should be centralized.** Implementation discoveries (Better-Auth deviation, test patterns, design system rules) should flow into a shared document. Story dev notes should reference, not repeat.

3. **Epic 1's brownfield discovery cost is Epic-1-specific.** Epic 2 is mostly greenfield (chat, AI parser, SSE streaming, alarm editor). Discovery overhead will be lower but new technology learning curves will appear.

4. **Story 1.0 infrastructure investment paid compound returns.** Error framework, shared types, mock datasource — used by every subsequent story without modification.
