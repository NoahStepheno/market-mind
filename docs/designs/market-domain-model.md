# market — 领域模型总览（V1）

**版本**：与 `docs/prds/market-prd.md` V1 对齐  
**范围**：各业务域的划分、职责边界与域间协作；告警域表级细节以 `docs/designs/market-alarm-domain.md` 为准。

---

## 1. 设计原则

| 原则             | 说明                                                                                                                 |
| ---------------- | -------------------------------------------------------------------------------------------------------------------- |
| 前期避免过度拆分 | V1 **不**将「触达」独立为单独限界上下文；推送的持久化与组织方式由**告警域**承担，用一张 `notifications` 表承载即可。 |
| 确认即落告警     | 用户未在聊天流中完成「确认创建」前，**不存在**持久化的告警实例（无 `alarms` 行）。                                   |
| 单用户数据隔离   | 跨域实体均带 `user_id`（或由鉴权上下文注入）；列表与详情默认按当前用户过滤。                                         |
| 可演进           | 若未来出现多通道运营、投递对账、模板版本化等需求，再将「触达」从告警域中**横向切出**，而不必推翻 V1 的核心边界。     |

---

## 2. V1 限界上下文划分

V1 采用 **三个** 业务域（原「用户权限 / 聊天 / 告警 / 触达」四块中的「触达」并入告警域）。

```text
┌──────────────────────────────────────────────────────────────────┐
│ 身份与访问（Identity & Access）                                        │
│  用户主体、登录鉴权、设备推送 token（若独立表）                               │
│  为其他域提供「当前 user_id」与鉴权失败语义                                 │
└───────────────────────────────┬──────────────────────────────────┘
                                │ 鉴权上下文
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
┌───────────────┐     ┌─────────────────────┐     ┌─────────────────────┐
│ 聊天域          │     │ 告警域（含 V1 触达）      │     │ 行情 / 指标（依赖）    │
│ 会话、用户 prompt │     │ alarms、评估、冷却、队列   │     │ 可为外部或自研模块     │
│ AI 草稿、确认前 UI │     │ notifications 表       │     │ 告警域只消费 Tick/衍生 │
└───────┬───────┘     └──────────┬──────────┘     └──────────┬──────────┘
        │ 用户「确认创建」           │ ← 行情                   │
        └──────────────────────────►│                         │
                  写入 alarms        │ 触发 → 写/发 notifications │
```

| 域             | 核心职责                                                                             | V1 持久化要点                                                                               |
| -------------- | ------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| **身份与访问** | 用户是谁、是否已登录、请求归属谁；可选：设备与推送 token                             | `users` 等；与 PRD 中「`userId` 仅服务端从鉴权注入」一致                                    |
| **聊天域**     | 会话、自然语言输入、**系统生成的结构化草稿**（不可直接当可执行告警）；确认流 UI 状态 | 会话与消息（含 prompt）、草稿快照；**不在告警域**存 `alarm_drafts` 或等价表（见告警域设计） |
| **告警域**     | 已确认的 `AlarmSpec`、规则评估、边沿与冷却、触发队列、**通知落库与投递组织**         | `alarms`、可选 `alarm_feedback`、**`notifications`**（见 §5）                               |

**命名说明**：若产品口语中仍说「用户权限域」，工程上 V1 可等价理解为 **身份与访问**（权限故事很薄时不必单独造势）。

**AI 解析**：不单独成域；作为**聊天域**内的能力（NL → 结构化草稿），边界仍是「只产出草稿，不写 `alarms`」。

**行情 / 指标**：可作为基础设施或单独模块；告警域**消费**已计算的标的维度数据，本总览不展开采集与聚合实现。

---

## 3. 聊天域：内容与生命周期

| 内容            | 含义                                                                         | 建议持久化策略                                                                                                                                                                              |
| --------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **用户 prompt** | 用户原话                                                                     | 若需会话回看、多轮修改，作为会话中的一条用户消息存储                                                                                                                                        |
| **系统草稿**    | AI 解析得到的结构化条件（PRD 中间态），**未经用户确认**                      | 挂在会话/助手消息或草稿字段；**确认前不写入** `alarms`                                                                                                                                      |
| **确认信息**    | 用户在结构化 UI 上调整后的最终条件，及冷却、`notify_label`、`notify_tier` 等 | **产品层**：可在聊天流记一条「用户已确认」的摘要；**技术层**：与「写入 `alarms`」为同一业务时刻——告警域写入为准，聊天侧不必再镜像完整 AlarmSpec，除非产品要强对比「当时说了啥 vs 最终规则」 |

**与告警域的接口**：用户完成确认后，由应用层调用告警域 **`POST /alarms`**（或 BFF 聚合），请求体来自确认 UI 的最终结构；`user_id` 来自 token，不可由客户端伪造。

---

## 4. 告警域：含 V1 触达（不单独划域）

V1 约定：**所有触达相关的组织与落库**均在告警域内完成，不引入独立的「触达限界上下文」。

- 规则、评估、冷却、队列载荷：`docs/designs/market-alarm-domain.md`（§6.2、§6.3）。
- 投递执行（查设备 token、调用 APNs/FCM 等）可实现为告警域内的**模块或服务**，对外仍属同一产品边界。

**内部「最小载荷」**（引擎 → 通知模块）：仍建议保持字段尽量小（`alarm_id`、`user_id`、`symbol`、`notify_label`、`notify_tier`、渲染所需包、内部幂等键等），便于队列重试与日志；这是**实现分层**，不等于单独业务域。

---

## 5. 表：`notifications`（V1 约定）

在告警域内用一张表承载「已产生的一次触达」或「待发送任务」，避免过度设计，但保留清晰归属与后续拆分切口。

| 列                       | 类型        | 说明                                                                                         |
| ------------------------ | ----------- | -------------------------------------------------------------------------------------------- |
| `id`                     | UUID        | PK                                                                                           |
| `user_id`                | UUID        | NOT NULL；FK → `users.id`                                                                    |
| `alarm_id`               | UUID        | NOT NULL；FK → `alarms.id`                                                                   |
| `dedupe_key`             | text        | 可选但**强烈建议**；用于队列重试时避免重复推送（如 `alarm_id` + 时间桶或内部触发标识的哈希） |
| `title` / `body`         | text        | 已渲染或可渲染快照，满足 PRD §6.1 / §3.8；**不**在此冗余完整 `condition_group`               |
| `notify_tier`            | enum        | 与触发时告警一致或快照，便于历史通知展示与审计                                               |
| `status`                 | enum        | 如 `pending` / `sent` / `failed`；失败时可配合重试次数（可选列）                             |
| `created_at` / `sent_at` | timestamptz | 投递生命周期                                                                                 |

**纪律**：

1. **归属清晰**：每行必能关联到 `user_id` 与 `alarm_id`。
2. **不复制规则源**：展示用文案以快照或最小渲染包为主；规则真源仍在 `alarms`。
3. **幂等与状态**：至少区分待发送与已发送，避免重试导致用户被重复骚扰。

用户端「通知列表」可主要查询本表；与 PRD「反馈关联 `alarmId`」仍一致。

## 6. 域间依赖小结

| 从 → 到       | 依赖方式                                                                    |
| ------------- | --------------------------------------------------------------------------- |
| 聊天 → 告警   | 确认后创建 `alarms`；聊天域不持久化已确认规则的第二真源（除非产品明确要求） |
| 告警 → 身份   | 所有写操作注入 `user_id`；读 token / 设备列表                               |
| 告警 → 行情   | 只读 Tick / 衍生指标；不写回行情库                                          |
| 客户端 → 各域 | 经鉴权的 HTTP/WebSocket；跨域编排可由 BFF 完成                              |

---

## 7. 非目标（V1）

- 不单独建立「触达域」微服务或独立限界上下文文档。
- 不在总览中重复 `alarms` 全字段与软删除细则（见告警域设计）。
- 不在 V1 强制暴露用户可见的「触发流水号」；内部幂等键仅存于 `notifications` / 队列，与 PRD 一致。
