# MarketMind — 身份与访问域设计方案

**版本**：与 `docs/prds/market-mind-prd.md` V1、`docs/designs/market-mind-domain-model.md` 对齐  
**范围**：谁可访问系统、请求归属哪个用户、与推送相关的设备标识；**不**展开聊天域、告警规则与通知正文业务。

---

## 1. 目标与原则

| 原则               | 说明                                                                                                                                   |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| **主体单一真源**   | 登录用户的主键（`user_id`）与凭证、会话由 **Better-Auth** 管理；业务表仅外键引用同一 `users.id`（或与 BA 默认用户表 ID 对齐的 UUID）。 |
| **服务端注入归属** | 与 PRD 一致：`alarms`、`notifications`、反馈等写入路径中的 `user_id`**仅来自鉴权上下文**，忽略客户端冒充字段。                         |
| **薄权限**         | V1 无多租户、无复杂 RBAC；「能调用 API」即已认证用户对本账号数据的访问。                                                               |
| **与告警域协作**   | 告警域依赖「当前用户」与可选「推送设备 token 列表」；表结构归属本域，告警域只消费（见总览 §2、告警域 §6.4）。                          |

---

## 2. 域边界

```text
┌─────────────────────────────────────────────────────────────┐
│ 身份与访问（本文档）                                              │
│  Better-Auth：注册/登录/OAuth、会话、Cookie 或 Bearer 策略            │
│  可选：push_device_tokens（或等价）→ 供通知模块解析 user_id → 设备      │
└───────────────────────────────┬─────────────────────────────┘
                                │ 鉴权上下文（user_id、401/403）
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
   聊天域                    告警域                    客户端
```

| 能力                                              | 归属                                                    |
| ------------------------------------------------- | ------------------------------------------------------- |
| 邮箱/社交登录、会话创建与失效、密码重置（若启用） | **Better-Auth** 默认能力与路由                          |
| `POST /alarms` 等业务 API 的认证中间件            | 本后端：**校验 BA 会话/JWT**，注入 `user_id`            |
| `alarms.user_id` → `users.id`                     | 外键约定在告警域表设计；用户行由 BA 迁移/表定义保证存在 |

---

## 3. Better-Auth 承载的核心功能

以下**不**在本域重复实现协议细节，以 [Better-Auth](https://www.better-auth.com/) 配置与插件为准（工程见 `apps/backend/src/modules/auth/`）：

- **身份建立**：OAuth（如 Google）等已选方式的用户创建与账号关联。
- **会话**：签发的会话或令牌校验；登出、过期策略。
- **用户存储**：BA 适配器对应的 `user` / `session` / `account` 等表（具体表名以项目迁移为准）。

本域文档只约定：**业务代码通过 BA 提供的 API（如 `auth.api.getSession`）解析请求**，得到稳定 `user.id`，再交给告警域、聊天域服务层。

---

## 4. 本域扩展（V1 可选但建议）

| 内容                         | 说明                                                                                                                                                                                         |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **推送设备表**（若独立出表） | `user_id`（FK → users）、平台（`ios` / `android` / `web`）、provider token、可选 `device_label`；**禁止**在业务请求体中单独信任「用户填的 userId」绑定设备，绑定动作必须在已认证会话下执行。 |
| **环境与密钥**               | `BETTER_AUTH_SECRET`、`BETTER_AUTH_URL`、OAuth Client 等仅运维/部署配置，不进产品协议。                                                                                                      |

若 V1 将推送 token 暂存于客户端本地、服务端仅按用户维度广播，可推迟设备表，但总览中「设备 token 列表」的归属仍算本域职责。

---

## 5. 对其他域的契约

| 消费者       | 契约                                                                                                                   |
| ------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **告警域**   | 所有 `/alarms*`、反馈、内部队列消费者：必须能解析 `user_id`；未认证 → 401；资源归属不符 → 404 或 403（产品一致即可）。 |
| **聊天域**   | 会话与消息按 `user_id` 隔离；草稿不写入 `alarms`，与总览一致。                                                         |
| **通知投递** | 队列载荷中的 `user_id` → 查本域（或 BA 用户 + 设备表）解析推送目标。                                                   |

---

## 6. API 形状（身份子集）

Better-Auth 挂载路径以实际路由为准（例如 `/api/v1/auth/*`）。**业务 REST**（告警、聊天）与 BA 路由分离：前者统一走同一鉴权中间件。

本域**不**在 V1 单独定义一套并行登录 API，避免与 BA 分叉。

---

## 7. 非目标（V1）

- 组织/团队、角色矩阵、细粒度策略引擎。
- 与 BA 并行的第二套用户表或自建密码哈希（除非未来明确迁移出 BA）。
- 在 PRD 范围外扩展「代用户外呼」等通道的身份强绑定（仍属产品扩展，非本 V1 文档核心）。
